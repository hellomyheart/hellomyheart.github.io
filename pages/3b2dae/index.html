# 图的第一步

图由顶点和边组成。边线作为文档存储在边线集合中。顶点可以是文档集合或边集合的文档（因此，边可以用作顶点）。通过边沿定义可在命名图形中使用哪些集合。命名图可以包含多个边缘定义，至少需要一个。图使您可以根据您的领域来构建模型，并在逻辑上将它们归类到集合中，并赋予您在同一图查询中查询它们的能力。

图形新手？参加我们的免费图谱课程以获取新生， (opens new window) 并从零知识到高级查询技术。

# 来自关系背景-什么是图表？

在SQL中，通常可以使用关系表的结构来存储两个数据表之间的n：m关系。一个边集合有点类似于这些关系表; 顶点集合类似于数据表和要连接的对象。虽然通过关系表在关系表中使用固定跳数的简单图形查询在带有多个嵌套联接的SQL中是可行的，但是图形数据库可以处理边缘集合上任意数量的这些跃点-这称为遍历。同样，一个边缘集合中的边缘可能指向多个顶点集合。通常将属性附加到边缘，即为这种互连命名的标签。边缘具有方向，与它们的关系_from并_to指出从一个文件到存储在顶点集合其他文档。在查询中，您可以定义沿哪个方向的边缘关系（OUTBOUND：_from→ _to，INBOUND：_from← _to，ANY：_from↔ _to）。

这里体现了关系型数据库和图形数据库的不同。

关系型数据库通过关系，将各个数据解耦成一个个表，通过表之间的主外键连接，完成查询。

通过关系拆分表，完成了数据的解耦，降低了冗余，但是由于DDL的限制，这些数据的格式也被限制死了，同时这些数据都在这些表里，操作某条数据的本质是操作表，通过限制条件在表中取得数据，然后通过这个数据做下一步操作。

图形数据库则是完成了表和数据（SQL数据库中的行数据）的解耦，

每一个数据元素都没有那些严格的DDL限制，每个元素（边或者顶点）都是相对独立的。

同时，由于这些元素相对独立，并且有边的概念，元素与元素之间进行关联，而不是表与表进行关联。进行一些递归操作，查询操作，查询各个元素的关系，更加方便(没有了表的限制)。

# 命名图

命名图完全由ArangoDB管理，因此也可以在Web界面中看到 (opens new window)。他们使用了ArangoDB图形功能的全部范围。您可以通过几个界面访问它们。

  • AQL图形操作

    具有以下几种风格：

    • 命名图和匿名图上的AQL遍历 (opens new window)
    • 命名图和匿名图上的AQL最短路径 (opens new window)

  • JavaScript General Graph实现，因为您可以在Foxx Services中使用它

    • 图管理 (opens new window); 创建和处理图形定义；在图中插入，更新和删除顶点和边
    • 图形功能， (opens new window)用于处理边和顶点，以分析它们及其关系

  • JavaScript SmartGraph实现，用于可伸缩图形

    • SmartGraph管理 (opens new window); 创建和操作SmartGraph定义；与一般图的差异

  • RESTful General Graph接口， (opens new window)用于在客户端驱动程序中实现图形管理

# 使用常规文档功能处理命名图的集合

可以使用标准的集合方法来访问命名图的基础集合。但是，graph模块在这些集合的顶部添加了一个附加层，为您提供以下保证：

  • 所有修改均以事务方式执行
  • 如果删除一个顶点，则引用该顶点的所有边也将被删除。
  • 如果插入边缘，则会检查边缘是否与边缘定义匹配

您的边缘集合将仅包含有效边缘，并且永远不会有松散的末端。如果您以除图形模块以外的任何其他方式访问集合，则这些保证将丢失，因此，如果直接从顶点集合中删除文档，指向它们的边将保留在原处。

创建命名图时，不会纠正数据中现有的不一致之处。因此，请确保从声音数据开始，否则可能会导致边缘悬空。图形模块保证不只引入新的不一致。

# 匿名图

有时您可能不需要命名图的所有功能，但其中的一些功能可能对您很有价值。您可以在遍历 (opens new window) 和“使用边” (opens new window)一章中使用匿名图。匿名图没有边缘定义来描述哪个顶点集合通过哪个边缘集合连接。图形模型必须在客户端代码中维护。与严格命名的图相比，这给您更多的自由。

  • AQL图形操作

    可用于命名图和匿名图：

    • AQL遍历 (opens new window)
    • AQL最短路径 (opens new window)

# 何时选择匿名或命名图？

如上所述，在插入或删除边或顶点时，命名图都可确保图的完整性。因此，即使您在多个命名图中使用相同的顶点集合，也不会遇到悬空的边缘。这涉及数据库内部的更多操作，这需要付出一定的代价。因此，匿名图在许多操作中可能会更快。因此，这个问题可能会缩小为：“我是否可以承担额外的努力，或者我需要完整性的保修？”。

# 多边缘集合vs边缘文档属性过滤器

如果只想遍历特定类型的边缘，有两种方法可以实现。第一是边缘文档中的属性-即type，在那里你指定的边缘微分-即"friends"，"family"，"married"或者"workmates"，让你可以在以后FILTER e.type = "friends" 如果你只是想跟着朋友边缘。

另一种方式，这可能是在某些情况下效率更高，是用不同的边缘集合了不同类型的边，让你有friend_edges，family_edges，married_edges和workmate_edges作为集合名称。然后，您可以配置几个命名图，包括可用的边和顶点集合的子集-或使用匿名图查询，在其中指定要在该查询中考虑的边集合的列表。要仅跟随朋友的边缘，可以将其指定friend_edges为唯一边缘集合。

两种方法都有优点和缺点。FILTER边缘属性上的操作将在每个遍历的边缘上进行比较，这可能会占用大量CPU资源。当因为包含它们的集合根本没有被遍历而根本找不到边缘时，将永远没有理由使用来实际检查它们的 type属性FILTER。

多边收集方法受到一个查询中可以同时使用的收集数量的 (opens new window)限制。查询中使用的每个集合都需要ArangoDB内部的一些资源，因此数量受到限制以限制资源需求。您可能还对其他边缘属性有约束，例如具有唯一约束的哈希索引，这要求文档在单个集合中以确保唯一性，因此可能无法在其中存储不同类型的边缘。多个边缘集合。

因此，如果您的边缘有大约十二种不同的类型，则可以选择收集方法，否则，FILTER首选该方法。当然，您仍然可以FILTER在边缘使用操作。您可以使用前一种方法摆脱对其他方法的FILTER依赖type，其他所有方法都可以保持不变。

# 我的数据的哪一部分是边缘，哪个是顶点？

数据模型中的主要对象（例如用户，组或文章）通常被视为顶点。对于每种类型的对象，文档集合（也称为顶点集合）应存储各个实体。实体可以通过边连接，以表达和分类顶点之间的关系。每个关系类型都有一个边集合通常是有意义的。

ArangoDB不需要您将数据存储在具有边和顶点的图结构中，您还可以决定_id在其他文档中嵌入属性（例如用户属于哪个组或文档的多个属性），而不是将文档与边连接。如果您的数据不关注关系并且不需要深度变化的图遍历，那么对于1：n关系而言，这可能是有意义的性能优化。如果嵌入数据，通常意味着引入一些冗余和可能的不一致，但这是可以接受的折衷方案。

这里牵扯到了属性的归属问题，对象元素的属性，如果有可能冗余，可以把这些元素变成一个新的文档（或者顶点），通过主外键或者边进行关联。

这两者之间如何取舍，由实际情况决定。

# 顶点（Vertices）

假设我们有两个顶点集合，Users和Groups。Groups集合中的文档包含组的属性，即组的成立时间，主题，图标URL等。Users文档包含特定于用户的数据-例如所有姓名，生日，头像URL，兴趣爱好…

# 边（Edges）

我们可以使用边集合来存储用户和组之间的关系。由于多个用户可以处于任意数量的组中，因此这是m：n关系。可以UsersInGroups用_from指向Users/John和_to指向的一条边来调用边集合Groups/BowlingGroupHappyPin。这使用户John 成为Bowling Group Happy Pin组的成员。此关系的属性可能包含此关系的限定词，例如John在该组中的权限，他加入该组的日期等。

粗略地说，如果您在句子中使用文档及其属性，则名词通常是顶点，动词会成为边缘。您可以在下面的已知图 (opens new window)中看到这一点：

# 这种方法的优点

图的优点是不仅可以在一行中具有固定数量的m：n关系，而且还可以具有任意数量的关系。可以在两个方向上遍历边缘，因此很容易确定用户所在的所有组，而且还可以确定某个组具有哪些成员。用户也可以互连以创建社交网络。

使用图数据模型，处理具有大量关系的数据将保持可管理性，并且可以通过非常灵活的方式进行查询，但是在关系数据库系统中处理它会引起头痛。

# 备份还原

为了确保您想拥有你的图形数据的备份，你可以使用Arangodump (opens new window)创建备份，并Arangorestore (opens new window)恢复备份到一个新的ArangoDB。但是，您应该注意：

  • _graphs如果备份命名图，则需要系统集合。
  • 您需要备份图形所组成的所有边缘和顶点集合的完整集合。部分转储/还原可能不起作用。

# 世界图

世界国家图按如下方式构造其节点：世界→大陆→国家→首都。在某些情况下，边缘方向不是向前的（因此它将在图形查看器中不连续地显示）。它有两种创建方法。一种使用命名的图形实用程序（worldCountry），另一种不使用（worldCountryUnManaged）。它用于演示原始遍历操作。

arangosh> var examples = require（“ @ arangodb / graph-examples / example-graph.js”）;
arangosh> var g = examples.loadGraph（“ worldCountry”）;
arangosh> db.worldVertices.toArray（）;
arangosh> db.worldEdges.toArray（）;
arangosh> examples.dropGraph（“ worldCountry”）;
arangosh> var g = examples.loadGraph（“ worldCountryUnManaged”）;
arangosh> examples.dropGraph（“ worldCountryUnManaged”）;

# Mps图

创建该图的目的是演示最短路径算法的用例。即使该算法只能确定一条最短路径，也可以通过两个单独的查询返回多个最短路径。因此，曲线图的得名米ultiple p ATH小号操作搜索 (opens new window)用例。

的示例曲线图由顶点中mps_verts收集和边缘中mps_edges收集。它是一个简单的遍历图形与起始节点甲和端节点Ç。

这是我们创建它，检查其顶点和边缘并再次将其放下的方式：

arangosh> var examples = require（“ @ arangodb / graph-examples / example-graph.js”）;
arangosh> var g = examples.loadGraph（“ mps_graph”）;
arangosh> db.mps_verts.toArray（）;
arangosh> db.mps_edges.toArray（）;
arangosh> examples.dropGraph（“ mps_graph”）;
